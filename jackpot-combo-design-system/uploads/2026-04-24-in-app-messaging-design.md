# 站內信系統設計（In-App Messaging）

- **版本**：v0.1（初版 brainstorm 成果）
- **日期**：2026-04-24
- **範圍**：僅站內信（站外信 Email / SMS / Push 為另一主題）
- **狀態**：待 review

---

## 1. 概述

站內信系統負責將**交易通知、活動優惠、玩家關懷、合規通知、系統公告**這五類訊息，透過站內訊息中心送達給玩家。本系統為**單向廣播型**，不包含玩家與客服的雙向對話。

訊息為純資訊性質，不附帶可領取的紅利 / 獎勵（MVP 範圍內）。

---

## 2. 目標與範圍

### 2.1 MVP 目標

1. 提供玩家一個統一、分類清晰、即時的訊息中心
2. 提供營運一套可手動發送、可自動觸發、可撤回、可稽核的發信後台
3. 從 day 1 收齊訊息送達 / 已讀 / CTA 點擊數據，支援合規查詢、客服查單、日後儀表板
4. 架構能從目前規模成長到未來大型量級（不過度設計，但不堵死擴充）

### 2.2 明確不做（非 MVP 範圍）

- 多語系（之後擴充）
- 多品牌 / 白標
- 玩家端刪除信件
- 訊息附帶可領取獎勵（紅利券 / 免費旋轉等）
- 發信雙人審核流
- 後台儀表板 UI（資料收齊，但不做圖表頁）
- 週期性排程（每週同一時間自動發）
- 客服與玩家雙向對話（另一個系統）
- 後台範本管理 UI（事件觸發訊息用 config 管理）
- 後台 RBAC（MVP 單一管理員角色）
- 站外信（Email / SMS / Push）

## 3. 用詞定義

| 用詞 | 定義 |
|------|------|
| **訊息 / 信件** | 一則站內信本體，含標題、分類、內文、CTA 等 |
| **收件人模式** | 單一玩家 / 手動輸入多筆 / 條件組合 / 全站廣播 四種 |
| **事件** | 業務系統發生的特定動作（存款成功、KYC 通過等），會觸發自動發信 |
| **範本** | 事件自動發信的文案模板，含變數占位符（如 `{{amount}}`） |
| **已讀** | 玩家點開訊息詳情後的狀態 |
| **CTA** | Call-to-Action，訊息底部的行動按鈕（如「前往活動頁」） |
| **啟用 / 停用** | 已發訊息的玩家端可見性開關；停用即等同撤回（玩家端消失，歷史保留），可再啟用恢復 |

## 4. 功能區塊總覽

| # | 區塊 | 服務對象 | 主要能力 |
|---|------|---------|---------|
| 1 | **玩家收件匣**（前台） | 玩家 | 查看訊息、已讀/未讀、分類、CTA、即時收到新訊息、toast |
| 2 | **後台發信中心**（後台） | 營運 | 手動發信、草稿、排程、分批、啟用 / 停用（撤回）、發信紀錄 |
| 3 | **事件自動發信**（系統內部） | 系統 | 接收業務事件、套 config 範本、自動發送 |
| 4 | **訊息狀態數據層**（資料） | 合規 / 客服 / 日後儀表板 | 每封信對每玩家的送達/已讀/CTA 紀錄；整合既有 audit log |

## 5. 前台（玩家）功能

### 5.1 信箱入口與未讀提示

- 玩家登入後，主導覽列（建議頭像旁）顯示**信箱 icon**
- 有任何未讀訊息時，icon 上顯示**紅點**；無未讀時不顯示（不顯示具體數量）
- **紅點狀態透過 WebSocket 即時更新**，不需刷頁
- 點擊 icon 進入收件匣

### 5.2 收件匣列表

- 頂部 tab：**個人訊息｜活動優惠｜系統公告**（進入收件匣預設停在「個人訊息」）
  - **個人訊息** = 交易通知 + 合規通知
  - **活動優惠** = 行銷活動 + 玩家關懷
  - **系統公告** = 維護、版本更新、全站公告
- 列表每項顯示：分類 icon、標題、內文預覽前 50 字、收信時間、未讀點
- 排序：**最新在上**
- **只顯示最近 90 天的訊息**（超過 90 天的訊息不顯示在前台；後端歷史仍保留）
- 分頁：每頁 20 筆，滑到底載入更多

### 5.3 訊息詳情

- 顯示：標題、收信時間、分類標示、富文字內文（含圖片）、CTA 按鈕（若有）
- **點開即自動標記已讀**（回寫 `read_at`）
- CTA 點擊時回寫 `cta_clicked_at`，並跳轉指定 URL
- **不提供刪除按鈕**

### 5.4 即時到訊與 toast

- 玩家在線時，新訊息經 WebSocket 推送：
  - 信箱 icon 紅點立即點亮
  - **僅個人訊息類（交易通知、合規通知）會觸發 toast**（活動優惠、系統公告不跳 toast，避免打斷遊戲沉浸感）
- **Toast 行為**：
  - 右下角彈出，5 秒後自動消失
  - 點擊 toast 跳轉到該訊息詳情
  - 同時間最多顯示 3 個 toast；若短時間內有第 4 封以後的訊息到達，以「你有 N 封新訊息」摘要 toast 彙總，點擊跳到收件匣

### 5.5 內容格式

訊息內文支援：
- 粗體、連結、分段的簡單富文字
- 內嵌圖片（後台上傳或 URL）
- 單一 CTA 按鈕（可選）

不支援自由 HTML（避免 XSS、RWD 難控、跨平台顯示不一致）。

## 6. 後台（營運）功能

### 6.1 建立訊息

營運進入「新增訊息」畫面，填寫：

- **標題**（必填）
- **分類**（必選：個人訊息 / 活動優惠 / 系統公告）
- **內文**（簡單富文字編輯器，可插入圖片）
- **CTA**（可選：啟用後填「按鈕文字」+「跳轉 URL」）
- **是否觸發 toast**（勾選框；預設值建議依分類，個人訊息預設開啟）

隨時可按「**存草稿**」，草稿列表頁可回來繼續編輯。

### 6.2 選擇收件人

建立訊息內文後，選擇收件人，四選一：

| 模式 | UI | 備註 |
|------|----|----|
| **單一玩家** | user ID / 帳號搜尋框 | 最常用於客服補發 |
| **手動輸入多筆** | 文字輸入框，以**逗號或換行**分隔多個 user ID | 送出前系統顯示「共 N 筆，其中 M 筆有效」 |
| **條件組合** | 從條件清單選欄位 → 選運算子 → 填值，可 AND / OR 組合 | MVP 支援下列 4 種條件 |
| **全站廣播** | 勾選「所有玩家」 | 送出前需二次確認 |

選擇完後**即時顯示預估發送人數**。

**MVP 支援的條件欄位**：

1. **VIP 等級**（如 `VIP ≥ 3`）
2. **玩家狀態**（活躍 / 沉睡 / 流失；具體定義由業務端提供）
3. **首存狀態**（已首存 / 未首存）
4. **註冊時間範圍**（如「30 天內新註冊」）

> **說明**：目前四項條件皆可在發信當下即時查詢玩家資料計算，不另建 tag 系統。未來可擴充欄位（累計存款級距、偏好遊戲類型、最近活動參與等），不建議一次塞太多。

### 6.3 送出與發送時間

送出訊息時填寫：

- **發送時間**（必填）：
  - 預設「**立即發送**」
  - 也可選擇未來的日期 + 時間進行排程；排程中的訊息在發送時間到達前可修改內容或取消
  - **不支援週期性排程**（MVP 延後）
- **全站廣播類**在送出時需再跳一次確認視窗，顯示「即將發送給 XXX 人，確定？」

**分批發送**：大量訊息（例如 > 5000 人）由系統自動分批執行，避免瞬間尖峰。分批速率由系統設定合理預設值（實際由 tech lead 決定），後台不需配置。

### 6.4 啟用 / 停用（撤回）

每封訊息有一個**啟用狀態開關**，送出後即進入「啟用中」，可隨時切換：

| 訊息狀態 | 玩家端可見 | 可執行動作 |
|----------|----------|-----------|
| 草稿 | ✗ | 繼續編輯 / 刪除 |
| 排程中 | ✗ | 修改 / 取消排程 |
| **啟用中** | ✓ | 按「停用」切到已停用 |
| **已停用**（等同撤回） | ✗ | 按「啟用」切回啟用中 |

**操作細節**：

- 「停用」與「啟用」兩個動作**都需要二次確認**（影響所有收件人）
- 由「已停用」切回「啟用中」時，**不會再次觸發 toast 推播**（避免營運切換開關時騷擾到玩家）
- 玩家的已讀 / CTA 點擊狀態在啟用 / 停用間保持不變（原本已讀繼續維持已讀）
- `message_read_states` 歷史紀錄完整保留，不論訊息目前是啟用或停用
- 啟用 / 停用操作都進 audit log

### 6.5 發信紀錄

後台列出所有訊息（草稿 / 排程中 / 啟用中 / 已停用），可依以下條件篩選：

- 日期範圍
- 分類
- 發送人（後台帳號）
- 訊息狀態

點擊單筆可查看詳細資訊、分批進度（如「已發送 23,000 / 50,000」）、基本狀態統計。

### 6.6 權限

MVP 為**單一管理員角色**，所有後台帳號權限相同。RBAC 延後實作。

## 7. 事件自動發信

### 7.1 MVP 事件清單（12 個）

| # | 事件 | 觸發時機 |
|---|------|---------|
| 1 | 存款成功 | 金流回填成功 |
| 2 | 存款失敗 | 金流回填失敗或逾時 |
| 3 | 取款申請已提交 | 玩家送出提款 |
| 4 | 取款核准 / 已打款 | 財務或系統核准 |
| 5 | 取款被拒絕 | 財務駁回（附原因） |
| 6 | KYC 審核通過 | 風控完成 |
| 7 | KYC 審核拒絕 | 風控駁回（附原因） |
| 8 | 密碼變更成功 | 玩家自行修改或重設 |
| 9 | 首次登入歡迎 | 玩家首次登入 |
| 10 | VIP 等級升級 | 等級提升 |
| 11 | 存款流水達成 | 玩家完成存款所需流水倍數 |
| 12 | 促銷 / 活動流水達成 | 玩家完成活動 / 紅利流水要求 |

事件註冊機制由工程側設計，後續新增事件只需按同一機制擴充，不需大改。

### 7.2 範本定義格式

每個事件的 config 需定義：

- **事件 ID**（例如 `deposit.success`）
- **分類**（個人訊息 / 活動優惠 / 系統公告）
- **允許使用的變數清單**（由工程對接業務系統決定可提供哪些變數）
- **標題範本**、**內文範本**（使用 `{{variable}}` 占位符）
- **是否觸發 toast**
- **CTA**（可選：按鈕文字、跳轉 URL；URL 可含變數）

### 7.3 MVP 範本清單（給 RD 製作用）

以下為 12 個 MVP 事件的初版文案範本，**文字內容由企劃維護、可調整**，變數命名為建議值（實際以工程對接業務系統可取得為準）。

#### 1. 存款成功（`deposit.success`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`amount`、`currency`、`payment_method`、`transaction_time`、`new_balance`
- **標題**：您的存款已到帳
- **內文**：親愛的 {{username}}，您的存款 {{amount}} {{currency}} 已於 {{transaction_time}} 成功到帳，目前餘額為 {{new_balance}} {{currency}}。
- **CTA**：「前往遊戲大廳」→ `/lobby`

#### 2. 存款失敗（`deposit.failed`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`amount`、`currency`、`transaction_time`、`failure_reason`
- **標題**：您的存款未成功
- **內文**：親愛的 {{username}}，您於 {{transaction_time}} 進行的 {{amount}} {{currency}} 存款未能完成，原因：{{failure_reason}}。若有疑問請聯繫客服。
- **CTA**：「聯繫客服」→ `/support`

#### 3. 取款申請已提交（`withdrawal.submitted`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`amount`、`currency`、`withdrawal_id`、`submission_time`、`estimated_review_hours`
- **標題**：取款申請已送出
- **內文**：親愛的 {{username}}，您的 {{amount}} {{currency}} 取款申請（編號 {{withdrawal_id}}）已於 {{submission_time}} 送出，預計於 {{estimated_review_hours}} 小時內審核完成。
- **CTA**：「查看申請狀態」→ `/withdrawal/{{withdrawal_id}}`

#### 4. 取款核准 / 已打款（`withdrawal.completed`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`amount`、`currency`、`withdrawal_id`、`completion_time`
- **標題**：取款已完成
- **內文**：親愛的 {{username}}，您的取款申請（編號 {{withdrawal_id}}）金額 {{amount}} {{currency}} 已於 {{completion_time}} 成功打款，請查收您的收款帳戶。
- **CTA**：（無）

#### 5. 取款被拒絕（`withdrawal.rejected`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`amount`、`currency`、`withdrawal_id`、`rejection_reason`
- **標題**：取款申請未通過
- **內文**：親愛的 {{username}}，您的取款申請（編號 {{withdrawal_id}}）金額 {{amount}} {{currency}} 未能通過審核，原因：{{rejection_reason}}。若有疑問請聯繫客服協助。
- **CTA**：「聯繫客服」→ `/support`

#### 6. KYC 審核通過（`kyc.approved`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`approval_time`
- **標題**：身分驗證已通過
- **內文**：親愛的 {{username}}，恭喜您！您的身分驗證已於 {{approval_time}} 完成，現在可以享用全部平台功能。
- **CTA**：「開始體驗」→ `/lobby`

#### 7. KYC 審核拒絕（`kyc.rejected`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`rejection_reason`、`resubmit_deadline`
- **標題**：身分驗證未通過
- **內文**：親愛的 {{username}}，您的身分驗證未能通過，原因：{{rejection_reason}}。請於 {{resubmit_deadline}} 前重新提交正確資料。
- **CTA**：「重新提交」→ `/kyc`

#### 8. 密碼變更成功（`password.changed`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`change_time`、`ip_address`
- **標題**：您的密碼已更新
- **內文**：親愛的 {{username}}，您的帳戶密碼已於 {{change_time}} 變更（來源 IP：{{ip_address}}）。若非您本人操作，請立即聯繫客服凍結帳號。
- **CTA**：「聯繫客服」→ `/support`

#### 9. 首次登入歡迎（`user.first_login`）

- **分類**：活動優惠　**Toast**：否
- **變數**：`username`
- **標題**：歡迎加入！
- **內文**：親愛的 {{username}}，歡迎加入我們！完成首次存款可領取迎新禮，我們為您準備了豐富的活動與優惠，祝您遊戲愉快！
- **CTA**：「立即存款領取迎新禮」→ `/deposit`

#### 10. VIP 等級升級（`vip.upgraded`）

- **分類**：活動優惠　**Toast**：否
- **變數**：`username`、`previous_tier`、`new_tier`、`upgrade_time`
- **標題**：VIP 等級升級通知
- **內文**：親愛的 {{username}}，恭喜您！您的 VIP 等級已於 {{upgrade_time}} 由 {{previous_tier}} 升級至 {{new_tier}}，現在起可享有更優質的專屬禮遇。
- **CTA**：「查看 VIP 專屬權益」→ `/vip`

#### 11. 存款流水達成（`deposit.turnover_met`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`completion_time`
- **標題**：存款流水已達成
- **內文**：親愛的 {{username}}，您已於 {{completion_time}} 完成存款流水要求，資金現在可正常提款。
- **CTA**：「前往提款」→ `/withdrawal`

#### 12. 促銷 / 活動流水達成（`promotion.turnover_met`）

- **分類**：個人訊息　**Toast**：是
- **變數**：`username`、`promotion_name`、`bonus_amount`、`currency`、`completion_time`
- **標題**：活動獎金已解鎖
- **內文**：親愛的 {{username}}，恭喜您完成「{{promotion_name}}」流水要求！獎金 {{bonus_amount}} {{currency}} 已於 {{completion_time}} 解鎖，可正常使用或提款。
- **CTA**：「前往提款」→ `/withdrawal`

### 7.4 範本管理方式

- **MVP 不做後台範本管理 UI**
- 範本以 **config 檔**（格式由 tech lead 決定）管理，存於程式碼版控中
- 營運需要修改文案時，透過工程流程（提 ticket → 改 config → 正常部署或 hotfix）
- 本平台部署週期為週，且支援 hotfix，工程側可在急迫情況下快速上線文案變更
- **未來若營運提案量大，可補一個僅能編輯文字欄位（標題 / 內文 / CTA）的輕量編輯畫面**，事件與變數定義仍維持在 code

### 7.5 失敗與重試

| 情境 | 行為 |
|------|------|
| 事件觸發但系統當下無法寫入訊息（DB 異常等） | 系統自動重試若干次；全失敗進「失敗佇列」，後台可查並手動重發。玩家不會收到訊息，但**業務狀態不受影響**（存款 / 提款本身已完成） |
| 同一事件被觸發多次（例如上游 retry） | 以事件 ID 去重，**同一事件 ID 只發一封訊息** |
| 訊息已寫入收件匣但 WebSocket 推送失敗（玩家離線） | 訊息仍在收件匣，玩家下次登入或重連時見到未讀。**不重送 WS 推送** |

**玩家是否會被通知「事件發信失敗」？**

不會。站內信 MVP 定位為通知管道，業務流程不依賴其送達保證。**需要保證送達的訊息（KYC 結果、取款狀態等）應走站外信（Email / SMS）**，屬下一階段主題。

## 8. 資料與合規

### 8.1 訊息狀態記錄

從 day 1 就收齊的紀錄，涵蓋每個玩家對每封訊息的狀態：

- 玩家 ID
- 訊息 ID
- 送達時間
- 已讀時間（null = 未讀）
- CTA 點擊時間（null = 未點 / 無 CTA）
- 訊息即使被停用（撤回），狀態歷史完整保留

**用途**：
1. 客服查單：玩家詢問「我沒收到某通知」時，可快速查詢
2. 合規查詢：回應監管方對特定訊息送達 / 已讀的查詢
3. 未來儀表板：直接以 SQL 計算開信率、CTA 轉換率
4. A / B 測試分析：比較不同文案的點擊率

**MVP 不做查詢 UI**，僅確保：
- 客服系統可透過 user ID + 日期區間撈資料（未來串接客服工具）
- 合規人員可透過 SQL 或 read-only view 查詢

### 8.2 操作日誌（整合既有 audit log）

**不新建後台頁面**，所有後台發信操作送進平台**既有的 audit log 系統**。

發信中心需送入 audit log 的事件（最少涵蓋）：

- 建立 / 編輯 / 刪除草稿
- 送出訊息（含收件人模式、預估人數、訊息 ID、發送時間）
- 排程修改 / 取消
- 訊息「啟用」/「停用」切換

每筆事件至少記錄：**操作人、時間、操作類型、目標訊息 ID、關鍵快照（標題、分類、收件人模式）**。

## 9. 非功能面

### 9.1 規模設計原則

本平台希望架構能從目前規模成長到未來大型量級。**具體儲存模型、分庫分表、快取策略交由 tech lead 設計**，產品面的要求為：

- 全站廣播（大量同時發送）不得拖垮主站
- 訊息狀態紀錄的寫入模式需能支撐未來百萬級玩家規模
- 分批發送避免瞬時尖峰

### 9.2 即時通知機制

產品面要求：**新訊息到達時，在線玩家應即時看到信箱紅點亮起**，個人訊息類並跳 toast。

實作方式預期為 WebSocket，**具體由 tech lead 決定**（包括斷線重連、多實例廣播、missed message 補發等）。

### 9.3 安全性

- 富文字內文**不允許自由 HTML**，採用受控編輯器或 Markdown 白名單
- CTA 跳轉 URL 需驗證為內部路徑或白名單網域
- 圖片上傳需限制大小 / 格式（具體由 tech lead 設計）

## 10. 明確延後（非 MVP）

- 多語系支援
- 多品牌 / 白標隔離
- 玩家端刪除信件
- 訊息附帶可領取獎勵

- 後台儀表板 UI
- 週期性排程
- 後台範本管理 UI
- 客服雙向對話
- 後台 RBAC
- 站外信（Email / SMS / Push）
